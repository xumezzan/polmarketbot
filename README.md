# Polymarket News Trading Bot

Production-style MVP новостного торгового бота для Polymarket.

На текущем этапе в README зафиксирована архитектура системы из ЭТАПА 1:
- как проходит pipeline
- за что отвечает каждый модуль
- как движутся данные от новости до paper trade
- почему LLM в системе только advisor, а не decision-maker

## Цель проекта

Собрать понятную и объяснимую инженерную систему, которая:
- получает новости
- анализирует их смысл через LLM
- ищет подходящий рынок на Polymarket
- считает потенциальное преимущество (`edge`)
- пропускает решение через deterministic risk engine
- открывает только paper trades
- сохраняет всю историю решений в БД

Главная идея:

```text
LLM помогает интерпретировать новость,
но не принимает финальное торговое решение.
```

Финальное решение о допуске сделки принимает `risk_engine` по жёстким и понятным правилам.

## Архитектура Pipeline

```text
[News Source]
      |
      v
[news_fetcher]
      |
      v
[news_normalizer]
очистка / нормализация / дедупликация
      |
      v
[llm_analyzer]
структурированный verdict по новости
      |
      v
[market_client]
поиск подходящего рынка Polymarket
      |
      v
[signal_engine]
fair_probability vs market_price => edge
      |
      v
[risk_engine]
детерминированные проверки допуска
      |
      v
[paper_trader]
виртуальная сделка
      |
      v
[database + monitoring]
сохранение истории и объяснений
```

## Роли модулей

### `news_fetcher`
Получает новости из внешнего источника, например NewsAPI.

Что делает:
- забирает свежие статьи
- сохраняет сырой контент для дальнейшей обработки
- не принимает торговых решений

### `news_normalizer`
Приводит новости к единому виду перед анализом.

Что делает:
- очищает текст
- нормализует заголовок, дату и URL
- удаляет дубли по URL или хешу содержимого

### `llm_analyzer`
Отправляет очищенную новость в LLM и получает структурированный verdict.

Ожидаемый смысл ответа:
- `relevance` — насколько новость релевантна рынку
- `confidence` — насколько модель уверена в интерпретации
- `direction` — `YES`, `NO` или `NONE`
- `fair_probability` — оценка справедливой вероятности
- `market_query` — текстовый запрос для поиска рынка
- `reason` — краткое объяснение

Важно: этот слой не говорит "покупай". Он только интерпретирует новость.

### `market_client`
Ищет подходящий рынок Polymarket по `market_query` из verdict.

Что делает:
- получает список рынков
- ищет наиболее подходящий матч
- достаёт цену, ликвидность и статус рынка

Все внешние обращения к Polymarket должны идти через отдельный клиентский класс.

### `signal_engine`
Преобразует verdict и данные рынка в торговый сигнал.

Базовая формула:

```text
edge = fair_probability - market_price
```

Результат:
- `REJECTED`
- `WATCHLIST`
- `ACTIONABLE`

Этот слой отвечает за простую и объяснимую математику сигнала.

### `risk_engine`
Самый важный модуль системы.

Он проверяет:
- confidence
- relevance
- свежесть новости
- ликвидность рынка
- дневной лимит
- дубли сделок

Возвращает:
- `allow` — можно ли открывать сделку
- `blockers` — список причин отказа
- `approved_size_usd` — допустимый размер позиции

Именно этот модуль принимает финальное решение о допуске сделки.

### `paper_trader`
Открывает и закрывает виртуальные позиции без реальных денег.

Что делает:
- открывает paper position
- закрывает её по правилам
- считает PnL
- сохраняет результат для статистики

### `repositories`
Изолируют работу с БД от бизнес-логики.

Примеры:
- `news_repo`
- `signal_repo`
- `trade_repo`

Это нужно, чтобы сервисы не писали SQL напрямую и чтобы слои оставались заменяемыми.

### `scheduler`
Запускает pipeline по расписанию, например раз в 15 минут.

Он должен только координировать порядок вызовов:

```text
fetch -> normalize -> analyze -> match -> signal -> risk -> paper trade
```

### `monitoring / logging`
Нужны для объяснимости.

Система должна отвечать на вопросы:
- почему новость была отброшена
- почему сигнал стал actionable
- почему risk engine заблокировал сделку
- почему позиция была закрыта

## Flow данных

Ниже упрощённый сценарий прохождения одной новости через систему.

```text
1. Пришла новость
   "Positive regulatory update for crypto market"

2. news_normalizer
   очищает текст и проверяет, не дубль ли это

3. llm_analyzer
   возвращает verdict:
   - relevance = 0.82
   - confidence = 0.76
   - direction = YES
   - fair_probability = 0.68
   - market_query = "bitcoin price"
   - reason = "positive regulatory signal"

4. market_client
   находит рынок Polymarket и получает market_price = 0.57

5. signal_engine
   считает edge = 0.68 - 0.57 = 0.11
   и помечает сигнал как ACTIONABLE

6. risk_engine
   проверяет свежесть, ликвидность, лимиты и дубли

7. paper_trader
   открывает виртуальную позицию

8. database
   сохраняет всю цепочку:
   новость -> анализ -> сигнал -> риск-решение -> paper trade
```

## Планируемые сущности в БД

### `news_items`
Хранит новости:
- источник
- заголовок
- URL
- текст
- дату публикации
- хеш содержимого

### `analyses`
Хранит результат LLM-анализа:
- relevance
- confidence
- direction
- fair_probability
- market_query
- reason

### `signals`
Хранит результат signal engine:
- market_price
- fair_probability
- edge
- signal_status
- explanation

### `paper_trades`
Хранит виртуальные сделки:
- сторону
- цену входа
- цену выхода
- размер позиции
- PnL
- статус

### `positions`
Хранит текущее состояние открытых виртуальных позиций.

## Ключевые инженерные принципы

### 1. LLM — advisor, не decision-maker
LLM помогает понять смысл новости, но не должна обходить жёсткие risk-правила.

### 2. Каждый слой заменяем
Должна быть возможность отдельно заменить:
- источник новостей
- LLM-провайдера
- клиент рынка
- paper trading на live trading

### 3. Все решения объяснимы
Нужно уметь восстановить цепочку:

```text
какая новость пришла
-> что сказала LLM
-> какой рынок нашли
-> какой был edge
-> какие risk-проверки прошли
-> почему открыли или не открыли позицию
```

### 4. Сначала paper trading
До реальных денег система должна пройти проверку на:
- качество сигналов
- корректность risk engine
- правильный расчёт PnL
- стабильность расписания

### 5. Сначала простой рабочий вариант
Без лишней сложности:
- без k8s
- без message bus
- без обучения собственной ML-модели
